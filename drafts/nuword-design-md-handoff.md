# DESIGN.md 引き継ぎ書 — NuWord向け

このドキュメントは、NuWordプロダクト（Seed / Verse / inner canvas）のClaude Codeに向けた引き継ぎ書です。
**DESIGN.mdとは何か**、**NOCTAでどのように作ったか**、**NuWordでどう作るべきか**を伝えます。

---

## 1. DESIGN.mdとは何か

> デザインルールをMarkdown1枚に収め、AIがそのまま読んでUI生成に使えるファイル。

もともとGoogleがデザインツール「Stitch」のアップデートで提唱し、世界中のデザイナー・開発者に広まった概念。Figmaのコンポーネントライブラリはビジュアルツール上にあり、AIが直接読んで実装に反映することが難しい。DESIGN.mdは**テキストだけ**なのでAIがそのまま読める。

### DESIGN.mdがあると何が変わるか

| 状況 | DESIGN.mdがない | DESIGN.mdがある |
|------|----------------|----------------|
| 新ページ追加 | 毎回「背景はクリームで、フォントはXで…」と指示が必要 | 「DESIGN.mdに従って」だけで一貫したUIが出る |
| 複数ページの一貫性 | Claudeが前のページを「思い出せない」 | 常に同じルールを参照できる |
| デザイン修正 | どこに何の色が使われているか把握しにくい | 1ファイルを直せば全体に反映できる |
| 引き継ぎ | 新しいセッションで振り出しに戻る | セッションをまたいでも同じ判断基準を持てる |

## 2. NOCTAでの作り方 — 実際にやったこと

NOCTAは音楽エンタメ会社のホームページ（`website/index.html`）を持っており、すでに独自のデザインシステムが実装されていた。DESIGN.mdを後付けで「発掘・文書化」するかたちで作成した。

### Step 1: コードを読んでブランドカラーを抽出した

`index.html` の Tailwind 設定ブロックを読み、カラートークンを列挙した。

```js
// index.html から抽出した例
colors: {
  brand: {
    bg:        '#05050F',   // ほぼ黒
    primary:   '#C8A2FF',   // ラベンダー
    secondary: '#C4942A',   // ゴールド
    vgreen:    '#39FF6A',   // ネオングリーン
  }
}
```

### Step 2: フォントを確認した

Google Fontsの `<link>` タグと `fontFamily` 定義を照合し、「どのフォントをどの役割に使うか」を整理した。

```js
// NOCTAの場合
font-display:  'Bebas Neue'      // ドラマチックな大数値
font-brand:    'Syne'            // ブランドロゴ・見出し
font-body:     'Inter'           // 英語本文
font-jp:       'Noto Sans JP'    // 日本語本文
```

### Step 3: CSSから「設計の意図」を読み取った

単に値を書き写すのではなく、「なぜこうなっているか」を解釈した。

例:
- `.glass` クラスを見て「グラスモーフィズムが設計パターンの中心」と判断
- `.reveal` アニメーションを見て「スクロールインがNOCTAの共通言語」と判断
- `body::before` のノイズテクスチャを見て「平坦さを防ぐブランドのサイン」と判断

### Step 4: インスピレーション源を決めた

NOCTAの既存デザインに「近い世界観のサイト」を複数挙げ、各サイトから学ぶポイントを明示した。

```
Apple      → 余白・ストーリーテリング
Linear     → ダークモードの精度
Vercel     → ミニマル・テクニカルアッシュ
Stripe     → セクション定義の明確さ
Lusion     → 創造的エージェンシー系ダーク
Pitch.com  → グラスモーフィズム
```

### Step 5: 「やってはいけないこと」を書いた

「やること」と同じくらい「やってはいけないこと」が重要。NOCTAの場合:

- ライトモードは使わない（ダーク専用）
- serif フォントは使わない
- グラデーションテキストを本文に使わない（読みにくい）
- z-index を 9999 以上にしない（ノイズレイヤーと競合）

### Step 6: 新規ページ作成用チェックリストを付けた

実際に使う場面を想定し、確認漏れを防ぐチェックリストで締めた。

### 完成ファイルの場所

```
/project_NOCTA/website/DESIGN.md
```

## 3. NuWordのDESIGN.mdを作るには

### NuWordのデザイン言語はNOCTAとは真逆

NOCTAは**暗・激・動**（ダーク・ネオン・アニメーション）。
NuWordは**明・静・余白**（ライト・オフホワイト・思索的）のはず。

実際のスクリーンショット（`website/images/seed.jpg` / `verse.jpg` / `inner-canvas.jpg`）を見ると:
- 背景はクリーム系のオフホワイト（inner-canvasは #FAF6F1 相当）
- テキストは落ち着いた暗色
- 余白が広く、情報密度は低い
- アニメーションは控えめ

参照すべきインスピレーション源の候補:

| サイト | 何を参考にするか |
|-------|---------------|
| **Notion** | クリーンな余白・構造化されたタイポグラフィ |
| **Bear / Craft** | ライトモードノートアプリの配色 |
| **Readwise** | 読書アプリ系のタイポグラフィ・暖色系背景 |
| **Things 3** | 静的で美しいUIのパターン |
| **Obsidian** | ナレッジツール系の設計思想 |
| **Apple Notes** | シンプル・余白・フォーカス |

### NuWordでのDESIGN.md作成手順

**Step 1: NuWordのソースコードを読む**

まず以下を確認する:
- `src/` または `app/` 以下の CSS/Tailwind 設定ファイル
- `tailwind.config.js` または `tailwind.config.ts`
- グローバルスタイルファイル（`globals.css` / `styles.css` など）

```bash
# 探すコマンド例
find . -name "tailwind.config*" -o -name "globals.css" | head -20
```

**Step 2: 3つのプロダクト共通の要素を抽出する**

Seed / Verse / inner canvas は同一ドメインにある。共通の設計要素があるはず:
- 背景色・テキスト色のトークン名
- 使っているフォント
- ボタンのスタイル
- カードのスタイル

**Step 3: 各プロダクトの「個性」を書き分ける**

3つが同じデザインシステムの上に乗りながらも個性を持つ場合、共通ルール＋プロダクト別セクションで書く:

```markdown
## 共通ルール
...

## Seed固有のルール
...（カード・リンク表現のルール）

## Verse固有のルール
...（テキスト引用の表現ルール）

## inner canvas固有のルール
...（AIチャット・リフレクション系のUI）
```

**Step 4: インスピレーション源のDESIGN.mdを参照する（任意）**

`getdesign.md` というサイトでApple・Notionなどのブランド向けDESIGN.mdがダウンロードできる。
参考として読むだけでよい（そのまま使うのはNG）。

## 4. DESIGN.mdを書く上で守ること

### AIが使える粒度で書く

「おしゃれな感じ」ではなく具体的な値で書く。

```markdown
<!-- NG -->
背景は温かみのある白

<!-- OK -->
背景色: #FAF6F1（クリームホワイト）。Tailwind なら bg-[#FAF6F1] または カスタムトークン bg-nuword-bg。
```

### 「なぜ」を書く

ルールの理由を書くと、AIが例外を判断できるようになる。

```markdown
<!-- NG -->
ゴシックフォントを使うこと

<!-- OK -->
本文フォント: Noto Sans JP。理由: NuWordは日本語ユーザーが主体で、
長文を読む場面が多いため可読性を優先。英字見出しには Inter を使ってよい。
```

### 「やってはいけないこと」セクションを必ず作る

最も使われやすい間違いを明示的に禁止することで、AIの暴走を防ぐ。

## 5. まとめ — NuWordのClaudeへのお願い

1. まず `tailwind.config.js` / `globals.css` を読んで既存のカラー・フォント定義を把握する
2. スクリーンショット（seed.jpg / verse.jpg / inner-canvas.jpg）を見て**「この画面はどんな設計哲学か」**を言語化する
3. インスピレーション源として Notion / Bear / Readwise あたりを参考に記述する
4. 共通ルール → プロダクト別ルール → やってはいけないこと → チェックリストの順で書く
5. ファイル名は `DESIGN.md` としてプロジェクトルートに置く

NOCTAのDESIGN.mdを参考として見たい場合:
`/project_NOCTA/website/DESIGN.md`（上記と同じリポジトリ内に存在する）
