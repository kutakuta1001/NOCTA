# 設計書: A/D scaffolding の軽量導入（CLAUDE.md）

日付: 2026-06-15
対象: `/Users/fghmacbook013/NOCTA/CLAUDE.md`（主ファイル）
出典: `project_NOCTA/harnes/opus-fable-scaffolding-case-study.html`（Opus 4.8 "Fable化"計画）

## 背景

Fable 5 が輸出管理ディレクティブで提供停止となり、fallback は Opus 4.8。
上記資料は「capability gap はプロンプトで埋まらないが scaffolding（足回り）は設計できる」として
4施策（effort-fit hook / A 多筋フレーミング / B 敵対的criticゲート / D Explore接地）を提示している。

Critic レビュー（Opus 4.8）の結論:
- 資料の中心施策 B は資料自身のドッグフードで根拠が大きく棄却された（fresh-context優位 3/3棄却・常時発火 3/3棄却）。
- NOCTA は創造主導・CEO審美判断・コスト規律のワークフローであり、コーディング正答率の gap は部分的にしか効かない。
- 全プロジェクトに注入する hook やコスト肥大する並列subagentパネルは NOCTA の規律（R-05・COST POLICY）と摩擦。

→ 低コストで NOCTA の既存ルールを強化できる **A（多筋フレーミング）と D（Explore接地）のみ**を、
   **コード・設計作業に限定**して軽量導入する。

## スコープ

- 適用対象: コード・設計作業のみ（HP の data 更新・SVP生成・hook/スキル改修）。創造フェーズ（歌詞・PV）は対象外。
- 配置先: `/NOCTA/CLAUDE.md` 主ファイル1本に集約。`project_NOCTA/CLAUDE.md`(v3.0) は触らない（二重管理回避）。
- 共有プラグイン（superpowers:brainstorming）は編集しない。常時コンテキストにある CLAUDE.md 記載により、brainstorming 起動時に自然に A/D が効く構造にする。

## 変更内容（2箇所）

### 変更1 — D（Explore接地）を R-05 に追記

R-05 の末尾に以下の趣旨を追記する:

> 既存コードベースで非自明な設計を行う場合は、実装前に Explore（または対象ファイル群の Read）で接地する
> — 関連サブシステム・既存パターン・命名規約・制約を把握してから設計する。
> ただし接地は「現状把握」であり既存パターンへの追従を強制しない（負債は改善対象として織り込む）。
> 自明な1-2ファイルの変更・新規ゼロのケースはスキップ可。

### 変更2 — A（多筋フレーミング）を新規サブセクションで追加

R-14 の直後に新規サブセクションを追加する（R番号は振らない。project_NOCTA 側 R-15 との衝突回避）:

> ### コード・設計作業の発散（多筋フレーミング）
> HP・SVP生成・hook/スキル改修など非自明なコード設計では、表層が違うだけの2-3案ではなく、
> 本質的に異なる2-3軸で案を出す（例: MVP優先 / リスク優先 / 制約優先）。
> 各軸が「何を最大化し・何を捨てるか」を明示する。設計生成はメインスレッドで行う（subagentに分散しない）。
> 自明な変更（1-2ファイルに閉じる・定型・明確指示）には適用しない。

## 意図的に入れないもの（YAGNI）

- effort-fit hook（2本）: 全プロジェクト注入の副作用があるため今回のスコープ外。将来 NOCTA限定発火を別途検討。
- 創造フェーズへの A 適用: スコープ判断（コード作業のみ）により対象外。
