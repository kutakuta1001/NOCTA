# Gyu 3D宝石 技術検討書 — 写真から本物の3Dジェムへ

作成日: 2026-07-05
起案: Fable 5（本検討書）／実装計画・実装: Opus 4.8 または Sonnet 5（CEO指示）
位置づけ: 実装計画の**前段の技術検討**。CEOの2つの問い（①3Dモデルの作成方法 ②宝石らしく見せる方法）に答え、推奨アーキテクチャを示す。

---

## 0. CEOの要望

- 反射されるものが単なる写真ではなく、**3Dモデルから作成した宝石のように見えるもの**
- それを**傾きに応じて遊べる**アプリにする
- 前提: 現行の「光にかざす」（kazasu.js・写真＋擬似光沢）のUI方向性は合っている

## 1. 検討① 3Dモデルの作成方法 — 4つの選択肢

| 方式 | 内容 | 長所 | 短所 | 判定 |
|---|---|---|---|---|
| **A. プロシージャル生成（コードでカット形状を生成）** | 宝石のカット（ブリリアント/オーバル/エメラルド/カボション等）は幾何学立体。Three.jsのコードでパラメトリックに頂点生成 | アセット0・36石を数種のカット×色で量産可・ライセンス問題なし・ページ重量最小・NOCTAの「アセットレス」制作と一致 | 実在標本の再現ではなく「理想化された宝石」になる | **採用** |
| B. Blender手作業 → glTF出力 | CEOまたはAIがBlenderでモデリング。KHR_materials_transmission/ior/dispersion拡張でマテリアルごと出力可 | 最高品質・自由度最大 | 36石分の制作工数・アセット重量・「実画像はgit管理外」規約との調整 | ヒーロー宝石1〜2個の将来強化用 |
| C. CC0モデルのダウンロード（Sketchfab/PolyHaven等） | 既製の宝石メッシュを利用 | 工数小 | ライセンス個別検証の負担（画像調達と同じ問題）・品質バラバラ・トポロジー汚い | 不採用 |
| D. AI 3D生成（Meshy/Tripo/TRELLIS等） | テキスト→3Dメッシュ | 話題性 | **宝石には不向き**: 宝石らしさは形状でなくマテリアル（屈折）にあり、AI生成メッシュは面が崩れてファセットの鋭さが出ない | 不採用 |

### 採用案Aの具体像 — カット5種のパラメトリック生成

宝石のカットは回転体＋平面ファセットの組み合わせで、コード生成に最適な形状:

1. **ラウンドブリリアント**（ダイヤ・ルビー・サファイア系）: ガードル多角形（16角）＋クラウン（テーブル面＋星型ファセット）＋パビリオン（尖った底）。頂点を手続き的に配置し `BufferGeometry` 化
2. **オーバル/クッション**: ブリリアントのガードルを楕円スケール
3. **エメラルドカット（ステップカット）**: 八角形の階段状。`ExtrudeGeometry`ベース＋面取り
4. **カボション**（オパール・ターコイズ・マラカイト・翡翠・ラリマー等の不透明/遊色石）: 半球ドーム（`SphereGeometry`の半分＋底面）
5. **結晶原石**（水晶系・蛍石・ベニトアイト等の標本らしさ）: 六角柱＋錐、またはConvexGeometryにランダム点

**重要**: 全カットで `flat shading`（面法線）必須。ファセットの境界がシャープに切り替わることで、傾けたときに面ごとに環境が「パチパチ切り替わる」= 宝石のきらめきの正体。

gems-data.js に3フィールド追加でマッピング:
```js
cut: "brilliant" | "oval" | "step" | "cabochon" | "crystal",
gem3d: "transparent" | "opaque" | "iridescent",   // マテリアルプリセット
ior: 2.42,   // 屈折率（ダイヤ2.42/サファイア1.77/水晶1.55/ガラス1.5…実物値をデータ化すると物語にもなる）
```

## 2. 検討② 宝石のように見せる方法 — マテリアルが本体

宝石らしさ＝**光学現象の再現**。優先順に:

| 現象 | 内容 | Three.jsでの実現 |
|---|---|---|
| **屈折** | 光が石内で曲がる（IOR 1.5〜2.4） | `MeshPhysicalMaterial` の `transmission:1` + `ior` + `thickness` |
| **ブリリアンス（内部全反射）** | パビリオン面で反射した光が正面に返る輝き | transmission材質のバックフェイス描画＋フラットシェーディングのファセットで近似 |
| **ファイア（分散）** | 波長で屈折率が違い、虹色に分かれる | `MeshPhysicalMaterial.dispersion`（three.js r167頃〜の公式サポート。**これが決め手**） |
| **色（吸収）** | 石の中を通る光が色づく | `attenuationColor` + `attenuationDistance`（accentColorから導出） |
| **表面反射・フレネル** | 角度で反射率が変わる | PBR標準（自動） |
| **遊色・虹彩**（オパール/ムーンストーン/アンモライト/ラブラドライト/パール） | 薄膜干渉 | `iridescence` + `iridescenceIOR`（MeshPhysicalMaterial標準機能） |

### マテリアルプリセット3種で36石をカバー

1. **transparent**（約24石）: `transmission:1, roughness:0, ior:石ごと, dispersion:0.1〜0.3, attenuationColor:accentColor` — ダイヤ・コランダム・ベリル系
2. **opaque**（約8石）: `transmission:0, clearcoat:1, roughness:0.15〜0.4, color:accentColor` — ターコイズ・マラカイト・ラピス・オブシディアン・翡翠
3. **iridescent**（約5石）: opaqueベース + `iridescence:1, iridescenceThicknessRange` — オパール・ムーンストーン・パール・アンモライト・ラブラドライト

### 環境マップ — きらめきの光源設計

宝石の輝きは**環境に何が映るか**で決まる。HDRファイルを置かず、**コードで環境シーンを構築**する:
- NOCTAのwarm-black（#0A0906）の暗い部屋に、細長い高輝度ライトストリップを3〜4本配置（宝石撮影のライトボックスと同じ考え方）
- three.js組み込みの `RoomEnvironment` をベースに調整、`PMREMGenerator` で前処理
- 暗環境×高輝度ストリップ → 傾けるとファセットに光の帯が走り「パチッ」と光る。NOCTAのダーク美学とも一致

### 品質を一段上げる小技（実装時のオプション）
- ごく弱いBloom（UnrealBloomPass）で最輝ファセットに滲み — モバイル性能次第でon/off
- カメラ僅かなドリー・宝石のゆっくり自転（idle時）— 触っていなくても生きている感
- 影: 石の下に接地影（現行kazasu.jsの `.gyu-shadow` を流用）

## 3. 実装アーキテクチャ（推奨）

```
website/gems/
├── gem3d.js        ← 新規: NoctaGem3d（ジオメトリ生成＋マテリアルプリセット＋レンダラ管理）
├── kazasu.js       ← 既存: 入力コントローラとして再利用（pointer/DeviceOrientationの正規化座標を提供）
└── index.html      ← ライトボックス内で 写真⇄3Dジェム のトグル
```

- **Three.jsの導入**: ビルドなし制約は `<script type="importmap">` + CDN（jsdelivr/unpkg）のESMで解決。Tailwind CDNと同じ整合性。three本体 ~170KB gzip。**遅延読み込み**（初回「光にかざす」時にdynamic import）でトップの表示速度に影響させない
- **入力**: kazasu.jsの正規化座標(-1〜1)ロジックを流用し、3Dでは宝石のクォータニオン回転＋慣性に接続。モバイルはDeviceOrientationで「傾けて遊ぶ」が本領発揮
- **描画**: 開いている間のみrAFループ（閉じたら停止・WebGLコンテキストは使い回し）
- **フォールバック**: WebGL不可/three読込失敗 → 現行の写真＋kazasu光沢に自動フォールバック（現行実装は削除しない）
- **写真との関係**: 実在標本の写真（出典付き）は図鑑としての価値なのでカード・キャプションに維持。3Dジェムは「その石の理想形を手に取る」体験として位置づけ、ライトボックス内にトグル（初期表示は3D）

## 4. リスクと対策

| リスク | 対策 |
|---|---|
| モバイル低速機でtransmission+dispersionが重い | dispersionのみ切る段階的品質・DPR上限1.5・開いている間だけレンダリング |
| three.jsのCDN依存（外部障害） | 読込失敗時は写真モードへフォールバック（3行のtry） |
| 「理想形の3D」と「実在標本の写真」の役割混同 | UIで「Photo / Gem」トグルを明示・キャプションの出典は写真側にのみ表示 |
| ページ重量 | dynamic import＋首回のみ取得。gems-data.jsへの追加は3フィールドのみ |

## 5. 次のステップ（Opus/Sonnetでの実装計画時に決めること）

1. カット5種の頂点生成の詳細設計（ブリリアントのファセット割りが最大の工数）

3. 環境ライトストリップの配置チューニング
4. Playwright検証項目（WebGL描画・トグル・フォールバック・リグレッション）
5. Phase分割: Phase 1=ブリリアント＋transparent材質＋5石で成立確認 → Phase 2=全カット・全36石 → Phase 3=Bloom等の磨き
